# CMIS 프로덕션 준비 작업리스트 (P0 / P1 / P2)

이 문서는 CMIS를 “프로덕션 수준(실사용 가능)”으로 끌어올리기 위해 필요한 **미구현/미완성 영역**을 정리한 작업 백로그입니다.

중요:
- 이 문서의 **P0/P1/P2는 ‘우선순위 분류’**입니다. **구현 순서를 강제하지 않습니다.**
- 실제 착수는 **리스크(데이터 손상/보안/비용)와 의존성, 사용자 학습 비용**을 기준으로 선택적으로 진행합니다.
- “인프라 변경(Postgres/S3 등) 자체”는 당장 하지 않더라도, **전환을 위한 준비 작업(계약/백엔드 인터페이스/마이그레이션 경로)**은 미리 닫아 둡니다.

---

## 0) 이 문서의 범위 정의

### 0.1 “프로덕션 수준”의 기준(최소)

- **단일 노드 운영(A 프로파일)**에서 장기 실행이 가능
  - `.cmis/` 기반 SQLite + ART 파일 저장이 **손상 없이 운영** 가능
  - 백업/복구/점검(doctor)으로 **운영 안전망**이 존재
- 실패/예외가 발생해도 **복구 가능한 실행 기록(run/ledger/trace)**이 남음
- 사용자가 실제로 사용해볼 수 있는 **최소 UX(명령/결과 탐색/검증)**가 존재

### 0.2 “프로덕션 수준”의 확장(목표)

- **팀 서버(B 프로파일)**: API 서버 + 워커 풀 + 중앙 SSoT(Postgres + Object Storage)
- **SaaS(C 프로파일)**: 로그인/권한/테넌시/리텐션/감사/egress 정책의 계약화 및 자동 검증

## P0 — 단일 노드 프로덕션 완결(하드닝/운영성/회귀 방지)

### P0-01 패키징/릴리즈/배포 기준선 고정

- **목표**: “설치/실행/업데이트”가 일관되게 동작하고, 버전/변경 이력이 운영 단위로 관리됨
- **미구현/미완성 포인트**
  - 설치형 패키징(예: `pyproject.toml`)과 CLI 엔트리포인트 표준화
  - 릴리즈 버전/체인지로그/호환성 정책
  - 최소 실행 환경/의존성 관리(락/취약점 관리 포함)
- **수용기준**
  - 새 환경에서 `cmis` 실행이 “문서대로” 재현됨
  - 버전이 출력/기록(run/trace)에 남음

### P0-02 CI(회귀 방지) 최소 파이프라인

- **목표**: “작은 변경이 시스템을 깨지 않도록” 자동 검증 기준선을 고정
- **미구현/미완성 포인트**
  - 테스트/린트/형식/설정(YAML) 파싱을 자동 실행하는 CI
  - 핵심 시나리오 스모크(브라운필드 E2E, Search v3 스모크, cursor doctor) 고정
- **수용기준**
  - PR/머지 단위에서 회귀가 자동 감지됨

### P0-03 스키마/계약 검증을 런타임에 연결

- **목표**: `/schemas/*.yaml`을 “문서”가 아니라 **Verifier/CI가 강제하는 계약**으로 승격
- **미구현/미완성 포인트**
  - ledgers/graphs/types의 스키마를 실제 저장(run/ledger/context/brownfield)에 대해 검증
  - schema drift를 탐지하고, 오류를 “원인 + 복구 가이드”로 보고
- **수용기준**
  - 잘못된 데이터가 저장되거나 export될 때 조기 탐지(doctor/verifier)

### P0-04 스토리지 운영 완성도(백업/복구/리텐션/락)

- **목표**: `.cmis/` 운영을 “실제 운영 절차”로 닫기
- **미구현/미완성 포인트**
  - 백업/복구 절차 및 자동화(스냅샷, 검증, 복구 리허설)
  - 리텐션(오래된 runs/artifacts 정리) 정책과 도구
  - 동시 실행/락 원칙을 명시하고 위반 감지(단일 writer 규칙 등)
  - reality_graph_store 운영 고도화(저장/공유/백업 전략 확정)
- **수용기준**
  - “데이터 손상 없이 오래 돌릴 수 있다”를 운영 도구가 뒷받침

### P0-05 캐시/영속 계층의 운영 프로파일 정리

- **목표**: evidence_cache / derived cache의 “정확한 성격(SSoT vs cache)”을 운영 옵션으로 고정
- **미구현/미완성 포인트**
  - evidence_cache의 기본 영속화 옵션 및 재시작 내구성
  - derived cache의 rebuild 규칙(언제 폐기/재생성하는지)
- **수용기준**
  - 캐시가 날아가도 시스템이 깨지지 않고, 원하면 유지도 가능

### P0-06 관측성(운영 로그/지표/비용) 최소 커널

- **목표**: 실패/지연/비용을 운영자가 “자동으로” 파악 가능
- **미구현/미완성 포인트**
  - run 단위의 요약 지표(성공/실패 이유, 외부 호출량, 대략적 비용)
  - LLM/API 사용량 및 비용(추정 포함) 집계/리포트
- **수용기준**
  - “무엇 때문에 실패했는지/얼마나 비용이 나갔는지”가 run 리포트로 즉시 확인 가능

### P0-07 Eval Harness 확장(회귀/카나리/지표 자동화)

- **목표**: 프로덕션 품질을 “테스트 가능/측정 가능” 상태로 유지
- **미구현/미완성 포인트**
  - 카나리(외부 API/검색 품질/LLM 응답 계약)와 회귀(핵심 시나리오) 분리
  - 결과 저장/비교 및 실패 시 원인 분석 자동화
- **수용기준**
  - 최소 1~2개의 “현실적인 시나리오”가 고정된 기준으로 계속 통과

## P1 — 실사용 UX를 만든다(사용자 입장에서 ‘써보고 개선’ 가능)

### P1-01 HTTP API 서버(Interaction Plane 확장)

- **목표**: CLI/커서 외에 “서비스 형태”로 접근 가능(워크플로 실행/조회/검증)
- **미구현/미완성 포인트**
  - API 서버(예: OpenAPI 계약, 요청/응답 스키마, 인증 전 단계라도 최소 라우팅)
  - run/ledger/artifact 탐색 API
- **수용기준**
  - 최소 3종 API: 실행 시작, run 상태 조회, 결과/아티팩트 조회

### P1-02 Web UI(결과 탐색/사용 로그 축적)

- **목표**: 사용자들이 “써볼 수 있는” 인터페이스를 제공하고, 로그/데이터가 쌓이도록 함
- **미구현/미완성 포인트**
  - run 탐색(결정/근거/아티팩트) UI
  - brownfield 입력/검증/커밋 UI
  - 사용자 행동/사용량 측정(기본 analytics)
- **수용기준**
  - 사용자가 UI에서 최소 한 번 end-to-end를 수행 가능

### P1-03 Search v3 Link Following 구현(설계 완료, 구현 예정)

- **목표**: SERP 결과의 “요약 페이지”를 넘어 IR/PDF/리포트 등 실질 근거로 도달
- **미구현/미완성 포인트**
  - `fetch_depth` + `link_selection` 기반 BFS 탐색
  - LinkExtracted/LinkFollowed 계보 이벤트 기록
  - budget/egress/SSRF 보호를 depth 탐색에도 동일 적용
- **수용기준**
  - authoritative phase에서 `fetch_depth=1`로 PDF/IR 링크를 따라가 evidence 품질이 실제로 개선됨

### P1-04 Brownfield 운영 UX 고도화(팀 내 재사용 관점)

- **목표**: “입력은 1회, 재사용은 여러 프로젝트”가 가능해지도록 UX/도구를 강화
- **미구현/미완성 포인트**
  - pack/템플릿/승인 흐름의 사용자 경험 정리
  - “명시적 입력” vs “프로젝트 진행 중 추가로 밝혀진 내용”의 분리 저장(옵트인/옵트아웃)
- **수용기준**
  - 사용자가 이전에 만든 brownfield 자산을 새 프로젝트에서 불편 없이 재사용 가능

### P1-05 LLM 모델 관리(설계 v1.1.0 기반 구현)

- **목표**: 모델 선택/에스컬레이션/비용/품질을 정책 기반으로 운영 가능하게 함
- **미구현/미완성 포인트**
  - ModelRegistry/TaskSpecRegistry/결정적 ModelSelector
  - 실패 유형 기반 escalation ladder
  - (최소) benchmark/회귀 감지의 기본 틀
- **수용기준**
  - 동일 effective_policy + 동일 입력이면 모델 선택이 결정적
  - 비용/사용량이 run 단위로 추적됨

## P2 — 멀티유저/팀 서버/권한/테넌시(서비스 운영의 본체)

### P2-01 로그인/회원가입/권한(RBAC)

- **목표**: 여러 사용자가 동시에 사용하는 서비스의 기본 전제 구축
- **미구현/미완성 포인트**
  - 계정/세션/권한 모델(누가 brownfield를 생성/수정/삭제 가능한지)
  - 관리자/편집자/뷰어 역할 분리
- **수용기준**
  - 최소 3권한: owner/editor/viewer가 실제로 정책에 의해 집행됨

### P2-02 사용자 그룹 기반 공유(팀/조직 단위 Brownfield 공유)

- **목표**: 같은 팀이 동일 SSoT(brownfield)를 공유하며 협업 가능
- **미구현/미완성 포인트**
  - 그룹/프로젝트 스코프에서 brownfield 자산(CUR/CUB/PRJ/BPK)을 공유하는 모델
  - 공유 범위/보안(외부 유출/egress)과 정책 결합
- **수용기준**
  - 그룹 내 공유/그룹 간 격리가 명확하고 검증 가능

### P2-03 SSoT 백엔드 전환 준비(Team Server 프로파일)

- **목표**: 단일 노드(A)에서 팀 서버(B)로 점진 전환 가능한 구조 확보
- **미구현/미완성 포인트**
  - Postgres 스토어 백엔드(런/레저/컨텍스트/브라운필드)
  - Object Storage(ART 바이트) 백엔드
  - 워커 풀/큐 기반 실행 모델
- **수용기준**
  - 로컬 SQLite를 유지하면서도 백엔드 교체 지점이 인터페이스로 닫혀 있음

### P2-04 사용량/비용 측정(빌링/쿼터/거버넌스)

- **목표**: 사용자 관점에서도 운영자 관점에서도 “비용을 통제”할 수 있게 함
- **미구현/미완성 포인트**
  - LLM/API 호출량/비용의 사용자·그룹 단위 집계
  - 예산 초과 방지(쿼터/경고/차단)
- **수용기준**
  - 사용자/그룹별 월간 비용 리포트가 자동 생성

## (부록) 향후 설계로 고도화하고 싶은 아이디어(초안)

아래 아이디어는 향후 **실질적인 설계 문서(계약/데이터 모델/정책/UX)**로 승격시키기 위한 “초안 메모”입니다.

### 1) Brownfield의 사용자별 개인화 및 범주별 분리 적용, 사용자 그룹화

- brownfield 입력은 사용 시작의 가장 큰 허들이므로, **한 번 입력한 brownfield를 다른 프로젝트에서 재활용**할 수 있어야 함
- 프로젝트 진행 중 추가로 설명되는 내용도 생기므로,
  - 사용자가 **명시적으로 입력한 brownfield**와
  - 프로젝트 수행 과정에서 **추가로 서술된 내용(암묵적/자동 수집 가능)**은
  - **분리 저장**되어야 함(사용자가 원하지 않을 수 있음)
- “cursor의 rules 저장”처럼 **사용자별 / 프로젝트별 / repo별**로 저장 계층을 나눌 필요가 있음
- 멀티유저 환경에서
  - **brownfield는 사용자별로 분리**
  - substrate plane의 graph(서비스 개선을 위한 지식)는 **전체 서비스가 공유**
  - 이 두 축을 함께 운영할 수 있는 구조가 필요
- 이를 위해 **로그인/회원가입**이 필요
- **사용자 그룹(팀/회사) 단위로 brownfield 공유**가 필요
- brownfield의 버전관리/생성/업데이트/삭제 권한을 통제할 수 있도록 **권한 관리(RBAC)**가 필요

### 2) WebUI 개발 필요 + 사용량 측정 모듈

- 다양한 사용자들이 실제로 사용하면서 데이터/로그가 쌓여야 실질적인 개선이 가능
- 사용자들의 사용량을 측정하는 모듈이 필요
  - 특히 외부 LLM/API 사용량과 비용을 사용자/그룹 단위로 측정하는 것이 중요
